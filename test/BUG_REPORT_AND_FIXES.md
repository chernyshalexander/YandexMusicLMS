## Краткая сводка

В плагине Yandex Music найдены две критические ошибки, которые нарушают корректную работу с несколькими аккаунтами и отображение форматов аудиофайлов. Обе ошибки имеют простое решение (замена функций и удаление строк кода).

## Ошибка #1: Неправильное использование аккаунта в ProtocolHandler

### Где найдена

**Файл:** `/home/chernysh/Projects/yandex/ProtocolHandler.pm`  
**Строки:** 159, 267, 534, 592

### Описание проблемы

Код использует функцию `getClient()` вместо `getAPIForClient($client)`. Это приводит к тому, что для **всех плееров всегда используется первый настроенный аккаунт**, независимо от того, какой плеер конкретно запрашивает данные.

Проблема возникает потому, что:

```perl
# Текущий код (НЕПРАВИЛЬНЫЙ):
my $api = Plugins::yandex::Plugin->getClient();  # Всегда первый аккаунт!
```

Функция `getClient()` внутри вызывает:
```perl
sub getClient {
    my $class = shift;
    return $class->getAPIForClient($_[0] || _getUserIdForClient(undef));
    #                                                          ^^^^
    #                                      undef — нет информации о плеере!
}
```

Когда `undef` передается в `_getUserIdForClient()`, функция возвращает первый пользователя из конфига.

### Как проявляется (симптомы)

1. **Радио (Rotor)** — Line 159  
   - У вас есть два плеера, настроенные на разные аккаунты Яндекс.Музыки
   - Второй плеер включает радиостанцию, но получает песни из профиля **первого плеера**
   - Рекомендации не соответствуют вкусам второго пользователя

2. **Потоковое вещание (Stream)** — Line 267 в `get_track_direct_url()`
   - Вторая сессия пытается включить трек
   - Используется токен потока от **первого аккаунта**
   - Может привести к ошибкам квоты или качества, если первый аккаунт исчерпал лимит

3. **Метаданные** — Line 534 в `getMetadataFor()`
   - Информация о треке (название, исполнитель, обложка) получается через первый аккаунт
   - Может быть недоступна для второго пользователя или быть устаревшей

4. **Раскрытие плейлистов** — Line 592 в `explodePlaylist()`
   - При раскрытии радиостанции используется информация первого аккаунта
   - Список песен может отличаться от того, что должен увидеть второй пользователь

### Когда возникает

- **Всегда**, когда в системе настроено больше одного аккаунта Яндекс.Музыки
- **Видно сразу** при попытке использовать второй плеер/аккаунт
- **Не влияет**, если использован только один аккаунт

### Как исправить

Замените все 4 вызова `getClient()` на `getAPIForClient($client)`.

#### Шаг 1: Line 159 (Rotor refill)

**Было:**
```perl
my $api = Plugins::yandex::Plugin->getClient();
```

**Стало:**
```perl
my $api = Plugins::yandex::Plugin->getAPIForClient($client);
```

#### Шаг 2: Line 267 (get_track_direct_url)

**Было:**
```perl
my $api = Plugins::yandex::Plugin->getClient();
```

**Стало:**
```perl
my $api = Plugins::yandex::Plugin->getAPIForClient($client);
```

Убедитесь, что переменная `$client` доступна в этом контексте (обычно она передается как параметр метода).

#### Шаг 3: Line 534 (getMetadataFor)

**Было:**
```perl
my $api = Plugins::yandex::Plugin->getClient();
```

**Стало:**
```perl
my $api = Plugins::yandex::Plugin->getAPIForClient($client);
```

#### Шаг 4: Line 592 (explodePlaylist)

**Было:**
```perl
my $api = Plugins::yandex::Plugin->getClient();
```

**Стало:**
```perl
my $api = Plugins::yandex::Plugin->getAPIForClient($client);
```

### Верификация исправления

После внесения изменений:

1. **Нужно проверить наличие переменной `$client`**
   ```bash
   grep -n "sub getMetadataFor\|sub explodePlaylist\|sub get_track_direct_url" ProtocolHandler.pm
   ```
   Каждый метод должен иметь `$client` в сигнатуре или в области видимости.

2. **Локальное тестирование (если доступна тестовая среда LMS)**
   - Настроить два аккаунта Яндекс.Музыки
   - Настроить их на двух разных плеерах (или эмулировать разные `$client`)
   - Включить радио на плеере #1 → должна быть рекомендация для пользователя #1
   - Включить радио на плеере #2 → должна быть рекомендация для пользователя #2
   - Проверить, что рекомендации отличаются (если у пользователей разные вкусы)

3. **Проверка других мест кода**
   ```bash
   grep -n "getClient()" ProtocolHandler.pm
   ```
   После исправления не должно быть вызовов `getClient()` без аргументов в этом файле.

## Ошибка #2: Жестко закодированное расширение .mp3 в Importer

### Где найдена

**Файл:** `/home/chernysh/Projects/yandex/Importer.pm`  
**Строки:** 24, 46, 375

### Описание проблемы

При импорте библиотеки Яндекса используется **только расширение `.mp3`**, независимо от того, какой кодек на самом деле использует трек. Это приводит к неправильному отображению формата и потенциальным проблемам с воспроизведением.

Проблема заключается в следующем коде:

```perl
# Строка 24 — объявление переменной
my ($ct, $splitChar);

# Строка 46 — установка жестко в mp3
$ct = 'mp3';  # ← ВСЕГДА mp3, никогда не обновляется!

# Строка 375 — использование в URL
my $url = 'yandexmusic://' . $trackId . '.' . $ct;
#                                      ^^^^^^^
#                            Всегда .mp3 добавляется!
```

### Как проявляется (симптомы)

1. **Неправильное отображение формата**
   - В библиотеке видна песня в формате FLAC, но LMS показывает как "MP3"
   - Песня в формате AAC отображается как "MP3"
   - Только после сканирования библиотеки и обновления кеша (3-5 минут) формат может измениться

2. **Потенциальные проблемы с воспроизведением**
   - Если плеер или клиент полагается на расширение для определения кодека, может выбрать неправильный обработчик
   - В лучшем случае это исправляется при воспроизведении, в худшем — трек не проигрывается

3. **Несоответствие с другими плагинами**
   - Плагины Yandex (Browse) и другие музыкальные сервисы правильно определяют кодеки
   - LMS показывает противоречивую информацию для одного трека через разные пути доступа

### Когда возникает

- **При первом импорте** библиотеки с командой сканирования
- **Видно сразу** в браузере LMS, когда смотрите информацию о треке
- **Особенно заметно**, если у вас смешанная библиотека (FLAC, AAC, MP3)

### Как исправить

Удалите расширение из URL и позвольте LMS определить кодек динамически через функцию `getFormatForURL()`.

#### Общий принцип

В LMS существует механизм определения формата файла по URL:
1. Если URL имеет расширение (`.mp3`, `.flac`), LMS использует его
2. Если расширения нет, LMS вызывает функцию `getFormatForURL()` плагина
3. `getFormatForURL()` в yandex плагине уже реализована и правильно определяет кодек из кеша

**Текущий способ в Browse/Common.pm (правильный):**
```perl
my $url = 'yandexmusic://' . $trackId;  # БЕЗ расширения
```

Это работает потому, что функция `getFormatForURL()` вызывается и правильно определяет кодек.

#### Шаг 1: Удаление расширения из URL

**Было (Line 375):**
```perl
my $url = 'yandexmusic://' . $trackId . '.' . $ct;
```

**Стало:**
```perl
my $url = 'yandexmusic://' . $trackId;
```

#### Шаг 2: Удаление неиспользуемого кода (опционально, но рекомендуется)

Проверьте, используется ли переменная `$ct`:

```bash
grep -n '\$ct' Importer.pm
```

Если `$ct` больше не используется нигде, удалите:
- **Line 24:** `my ($ct, $splitChar);` → `my ($splitChar);` (оставьте $splitChar, если она используется)
- **Line 46:** `$ct = 'mp3';` → удалите полностью

Если `$ct` используется в других местах, оставьте объявление и инициализацию, просто не используйте в URL.

### Верификация исправления

После внесения изменений:

1. **Проверьте синтаксис**
   ```bash
   perl -c Importer.pm
   ```

2. **Убедитесь, что нет других использований `$ct` в контексте URL**
   ```bash
   grep -n "'\.' \. \$ct\|'.' . \$ct\|\$ct" Importer.pm
   ```

3. **Тестирование (если доступна тестовая среда LMS)**
   - Полностью очистить кеш LMS
   - Запустить полное сканирование библиотеки Яндекса
   - Импортировать плейлист или альбом с смешанными кодеками (если доступны)
   - Проверить, что сразу (без задержки) показываются правильные форматы:
     - FLAC треки → показать как FLAC
     - AAC треки → показать как AAC  
     - MP3 треки → показать как MP3
   - Попробовать воспроизвести каждый трек в браузере LMS

4. **Проверка совместимости**
   - Убедиться, что функция `getFormatForURL()` в plibyin определяет все поддерживаемые кодеки:
   ```bash
   grep -A 20 "sub getFormatForURL" ProtocolHandler.pm
   ```
   Функция должна обрабатывать: `flac`, `aac`, `mp3`, `ymf`, `yma` и другие.

## Связь между ошибками

Обе ошибки возникли из-за **копирования старого кода без адаптации к текущей архитектуре плагина**:

- В **ProtocolHandler** когда-то использовался только один аккаунт, теперь есть поддержка нескольких
- В **Importer** была необходимость явно указывать формат; теперь есть динамическое определение через `getFormatForURL()`

---

## Порядок исправления

1. **Сначала исправьте ошибку #1** (ProtocolHandler с аккаунтами)
   - Это более критично и влияет на функциональность с несколькими аккаунтами
   - Изменения изолированы, не влияют на остальной код

2. **Затем исправьте ошибку #2** (Importer с расширением)
   - Может потребовать очистки кеша при тестировании
   - Менее критично, но улучшает UX

## Дополнительная информация для отладки

### Как проверить текущий аккаунт в методе

Если вы сомневаетесь, правильно ли передана переменная `$client`, добавьте отладку:

```perl
my $userId = Plugins::yandex::Plugin->_getUserIdForClient($client);
my $accountName = Plugins::yandex::Plugin->_getUserNameForClient($client) || 'Unknown';

main::INFOLOG && main::logger('info', "Using account for client: $accountName (userId: $userId)");
```

### Как проверить доступные функции в Plugin

```bash
grep -n "sub getAPIForClient\|sub getClient\|sub _getUserIdForClient" Plugin.pm
```

Убедитесь, что функции существуют и документированы.
